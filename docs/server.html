Server

The campfire has a built-in webserver within StreamingAssets that will run upon starting the program using the RunServer script, with a static library WSUtils within the Socket Manager script.

Message Format​

Messages are set up in the format:

Message
{
    eventName:[string],
    data:[string]
}

This ensures ease of sending over websockets but requires pre-processing to messages before sending and needing to decode messages upon retrieval. Within Javascript, the JSON.stringify() function can be used to easily convert messages into strings.

EventName​

The event name is used to differentiate different messages from one another, typically by use of an if logic statement or a switch statement before processing the data. In the case that a simple string is broadcasted, it is typically safe to assume that the string can be substituted in as the eventName.

Data​

Data is sent over as a string, which in it of itself can contain a multitude of different types through the JSON format. While it can commonly be used to store primitive types, it is also possible to convert other custom classes or data types into a string and store it as such.

Server Setup​

The server is setup in a way that it will pass through and broadcast any message sent to it to all other users connected. Typically, the server must be compiled into an .exe file that will be run upon starting the application.

Server.js​

The server uses express and websockets to handle connections and sending messages between clients. The server by default uses the :2000 port, and the public IP for connections. On the Campfire, this is: http://campfire3.idea.rpi.edu:2000/

When a client connects, it will log that connection to the console and then wait for any messages. When a message is received, it will attempt to parse it in the message format listed above. If it fails to do so, it will treat the entire message as the eventName instead.

Additionally, the server allows for the possibility of dividing users by ID, in order to only send messages to specific clients.

eventName:DataDescription:
set-idid (int)Sets a client to be associated with a specific ID. A single client can be associated with multiple IDs.
remove-idid (int)Removes a client from a specified ID.

By default, messages will send to every ID, in order to send a message to a specific ID, the event name will need to be prepended with [int]:[eventName].

{eventName:"1:exampleEvent",data:"Hello world."}

This will send out the event exampleEvent with the data Hello world. to all users associated with the ID 1.

Public​

The public folder holds all of the files that can be sent out to clients from scripts to HTML files that they will receive. The server will process any .html files found within, and it is possible to access any .html files within by appending /[filename] or /[filename.html] to the end of the URL. This does not look for any .html files within any subfolders, so they must be found directly within the public folder.

ServerUtils.js​

The serverutils.js file contains a simple method for connecting to the server with autoreconnection if the websocket is ever dropped. It will look for a websocket that matches the server url and port.
Upon the connection opening, it will call the OpenOpen(event) function, which can be defined as needed within the script of the website.
Upon receiving a message, it will call the OnMessage(event), which can be defined as needed within the script of the website.
If the socket loses connection, it will log to the console that the connection has failed, and reattempt a connection after a second.

Javascript Client Setup​

In order to use the serverutils.js script, it must be included within the <head> tag of the html file.

Using serverutils.js
<head>
    <script src="serverutils.js"></script>
</head>

Once it is included, it can be used within any script of the html file, either within <script> tags or within an included .js file.

Then, the functions OnMessage(event) and OnOpen(event) can be defined. In order to get the message as a string, it must be converted using event.data.toString("utf8"). For example:

On Message
function OnMessage(event) {
    let message = {eventName: "", data: ""};
    try {
        message = JSON.parse(event.data.toString("utf8"));
    }
    catch {
        return;
    }
    console.log(message);
}

Before using any server related feature, the client must first connect to the server using Connect(), otherwise it will receive or be able to broadcast any messages.

Unity Client Setup​

The Unity setup is very simple through the Socket Manager script. Most of the functions needed will be accessed using the WSUtils script.

using Campfire;

...

    private void Awake()
    {
        WSUtils.socketManager.OnMessage.AddListener(OnMessage);
    }

    private void Start()
    {
        WSUtils.socketManager.Connect();
    }

    private void OnMessage(WSResponse response)
    {
        Debug.Log(string.Format("Event Name: {0}\nData: {1}", response.eventName, response.data));
    }

For most use cases, these functions will be all that is necessary for use with the websocket server. The WSResponse class has two fields, and is setup as such:

WSResponse
{
    public string eventName;
    public string data;
}

In additional, the WSUtils will also handle reading and converting any messages properly into a WSResponse.

However, as C# within Unity does not easily support dynamically typed variables, it will be necessary to define any custom classes you wish to read before hand.
